**Тема:** Виявлення акторів і варіантів використання. Короткий опис варіантів використання

**Мета роботи:** Необхідно для автоматизованої інформаційної системи (веб-сайт для подачі документів абітурієнтами) визначити основних акторів і сформулювати варіанти використання, уточнити список варіантів використання, конкретизувати їх описи, скласти реєстр функціональних вимог.

## Актори і варіанти використання

### Актори та їх короткий опис

| Актор | Опис |
|-------|------|
| **Абітурієнт** | Основний користувач системи, який реєструється на сайті, подає документи для вступу, завантажує скан-копії документів, переглядає статус своєї заявки, отримує повідомлення від комісії, переглядає інформацію про факультет та спеціальності. Поділяється на зареєстрованого та незареєстрованого (гість). Незареєстрований абітурієнт має обмежений доступ лише до інформаційних сторінок. |
| **Працівник відбіркової комісії** | Користувач з розширеними правами, який має доступ до бази поданих документів, перевіряє документи абітурієнтів на коректність та повноту, змінює статус заявок (подано, перевіряється, прийнято, відхилено), надсилає повідомлення абітурієнтам, формує звіти за результатами вступної кампанії. |
| **Адміністратор системи** | Технічний фахівець, який налаштовує систему, управляє обліковими записами користувачів, надає права доступу, виконує резервне копіювання даних, моніторить працездатність системи, переглядає логи та статистику використання. |
| **Система ЄДЕБО** | Зовнішня державна система, з якою інтегрується веб-сайт для передачі даних про абітурієнтів та їх заявки відповідно до державних вимог. |

## Модифіковані схеми з відображенням відношень розширення та включення

### Відношення включення (include)

**1. Створення заявки включає вибір спеціальності:**
```
(Створення заявки) ──include──> (Вибір спеціальності)
```
Пояснення: Неможливо створити заявку без вибору спеціальності, тому це обов'язкова частина процесу.

**2. Завантаження документів включає перевірку формату:**
```
(Завантаження документів) ──include──> (Автоматична перевірка формату)
```
Пояснення: При завантаженні кожного документа система автоматично перевіряє його формат та розмір.

**3. Перевірка документів включає зміну статусу:**
```
(Перевірка документів) ──include──> (Зміна статусу заявки)
```
Пояснення: Після перевірки документів комісія обов'язково змінює статус заявки.

### Відношення розширення (extend)

**1. Відхилення заявки розширює перевірку документів:**
```
(Відхилення заявки) ──extend──> (Перевірка документів)
                      [документи некоректні]
```
Пояснення: Якщо під час перевірки виявлено помилки, може бути виконано відхилення заявки.

**2. Надсилання сповіщення розширює зміну статусу:**
```
(Надсилання email-сповіщення) ──extend──> (Зміна статусу заявки)
                                [автоматичне сповіщення увімкнено]
```
Пояснення: При зміні статусу заявки абітурієнт може автоматично отримати сповіщення на email.

**3. Повторне завантаження розширює видалення документа:**
```
(Повторне завантаження) ──extend──> (Видалення документа)
                         [потрібна заміна файлу]
```
Пояснення: Після видалення некоректного документа абітурієнт може завантажити новий.

## Результуючий реєстр функціональних вимог

**Абітурієнт:**
1. (A1) Система повинна надавати можливість абітурієнту зареєструватись, створивши обліковий запис з особистими даними.
2. (A2) Система повинна надавати можливість абітурієнту авторизуватись, використовуючи email та пароль.
3. (A3) Система повинна надавати можливість абітурієнту створити заявку на вступ з вибором спеціальності.
4. (A4) Система повинна надавати можливість абітурієнту завантажити скан-копії необхідних документів.
5. (A5) Система повинна надавати можливість абітурієнту переглядати поточний статус своєї заявки.
6. (A6) Система повинна надавати можливість абітурієнту редагувати свої персональні дані в профілі.
7. (A7) Система повинна надавати можливість абітурієнту переглядати інформацію про факультет та спеціальності.
8. (A8) Система повинна надавати можливість абітурієнту переглядати повідомлення від працівників комісії.
9. (A9) Система повинна надавати можливість абітурієнту видаляти раніше завантажені документи.

**Працівник відбіркової комісії:** <br/>
10. (K1) Система повинна надавати можливість працівнику комісії авторизуватись з службовими обліковими даними. <br/>

### 1. Реєстрація в системі (A1)

**Контекст використання:** Дає змогу абітурієнту створити обліковий запис для доступу до всіх функцій системи подачі документів.

**Зона дії:** Модуль управління користувачами.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Абітурієнт (незареєстрований).

**Учасники і інтереси:** 
- Абітурієнт - хоче швидко та зручно зареєструватись
- Система - повинна зберегти коректні дані користувача
- Університет - хоче отримати валідні дані абітурієнтів

**Передумова:** Абітурієнт перейшов на головну сторінку сайту і натиснув кнопку "Реєстрація".

**Мінімальні гарантії:** У разі помилки реєстрації всі введені дані зберігаються у формі, виводиться повідомлення про помилку.

**Гарантії успіху:** Створено обліковий запис, абітурієнт отримує email-підтвердження, система автоматично авторизує користувача.

**Тригер:** Натискання кнопки "Зареєструватися" на формі реєстрації.

**Основний сценарій:**
1. Абітурієнт натискає кнопку "Реєстрація" на головній сторінці.
2. Система відображає форму реєстрації.
3. Абітурієнт заповнює поля: прізвище, ім'я, по батькові, email, телефон, пароль, підтвердження пароля.
4. Абітурієнт погоджується з умовами обробки персональних даних.
5. Абітурієнт натискає кнопку "Зареєструватися".
6. Система перевіряє коректність введених даних.
7. Система перевіряє унікальність email.
8. Система створює обліковий запис.
9. Система надсилає лист з підтвердженням на вказаний email.
10. Система відображає повідомлення про успішну реєстрацію.

**Розширення:**
- 3а. Абітурієнт залишає обов'язкові поля порожніми:
  - 3а.1. Система підсвічує незаповнені поля червоним кольором.
  - 3а.2. Повернення до кроку 3.
- 6а. Формат email некоректний:
  - 6а.1. Система виводить повідомлення "Некоректний формат email".
  - 6а.2. Повернення до кроку 3.
- 6б. Пароль не відповідає вимогам (менше 8 символів):
  - 6б.1. Система виводить повідомлення про вимоги до пароля.
  - 6б.2. Повернення до кроку 3.
- 6в. Паролі не співпадають:
  - 6в.1. Система виводить повідомлення "Паролі не співпадають".
  - 6в.2. Повернення до кроку 3.
- 7а. Email вже зареєстрований у системі:
  - 7а.1. Система виводить повідомлення "Користувач з таким email вже існує".
  - 7а.2. Система пропонує перейти до авторизації або відновити пароль.
  - 7а.3. Закінчення варіанту використання.

### 2. Авторизація в системі (A2)

**Контекст використання:** Дає змогу зареєстрованому абітурієнту увійти в систему для доступу до особистого кабінету.

**Зона дії:** Модуль управління користувачами.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Абітурієнт (зареєстрований), Працівник комісії.

**Учасники і інтереси:**
- Користувач - хоче швидко отримати доступ до системи
- Система - повинна забезпечити безпечний вхід

**Передумова:** Користувач має активний обліковий запис у системі.

**Мінімальні гарантії:** У разі невдалої спроби входу облікові дані користувача залишаються незмінними, виводиться повідомлення про помилку.

**Гарантії успіху:** Користувач успішно авторизований, відкрито його особистий кабінет згідно з роллю.

**Тригер:** Натискання кнопки "Увійти" на формі авторизації.

**Основний сценарій:**
1. Користувач переходить на сторінку авторизації.
2. Система відображає форму входу.
3. Користувач вводить email та пароль.
4. Користувач натискає кнопку "Увійти".
5. Система перевіряє наявність користувача з таким email.
6. Система перевіряє відповідність пароля.
7. Система створює сесію користувача.
8. Система перенаправляє користувача до особистого кабінету відповідно до ролі.

**Розширення:**
- 5а. Користувача з таким email не знайдено:
  - 5а.1. Система виводить повідомлення "Невірний email або пароль".
  - 5а.2. Повернення до кроку 3.
- 6а. Пароль невірний:
  - 6а.1. Система виводить повідомлення "Невірний email або пароль".
  - 6а.2. Система збільшує лічильник невдалих спроб входу.
  - 6а.3. Повернення до кроку 3.
- 6б. Кількість невдалих спроб перевищено (більше 5):
  - 6б.1. Система блокує обліковий запис на 30 хвилин.
  - 6б.2. Система виводить повідомлення про тимчасове блокування.
  - 6б.3. Закінчення варіанту використання.

### 3. Створення заявки на вступ (A3)

**Контекст використання:** Дає змогу абітурієнту створити нову заявку на вступ із вказанням бажаної спеціальності та освітньої програми.

**Зона дії:** Модуль управління заявками.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Абітурієнт (авторизований).

**Учасники і інтереси:**
- Абітурієнт - хоче подати заявку на бажану спеціальність
- Відбіркова комісія - хоче отримати коректно оформлені заявки
- Університет - хоче систематизувати процес прийому заявок

**Передумова:** Абітурієнт успішно авторизований у системі, перебуває в особистому кабінеті.

**Мінімальні гарантії:** У разі помилки створення заявки дані не зберігаються, абітурієнт може повторити спробу.

**Гарантії успіху:** Заявка створена зі статусом "Чернетка", абітурієнт може додавати до неї документи.

**Тригер:** Натискання кнопки "Створити нову заявку" в особистому кабінеті.

**Основний сценарій:**
1. Абітурієнт натискає кнопку "Створити нову заявку".
2. Система відображає форму вибору спеціальності.
3. Система завантажує список доступних спеціальностей факультету.
4. Абітурієнт обирає спеціальність зі списку.
5. Система відображає список освітніх програм для обраної спеціальності.
6. Абітурієнт обирає освітню програму.
7. Система відображає інформацію про вимоги до документів.
8. Абітурієнт натискає кнопку "Підтвердити створення".
9. Система створює заявку зі статусом "Чернетка".
10. Система генерує унікальний номер заявки.
11. Система перенаправляє абітурієнта на сторінку редагування заявки.

**Розширення:**
- 3а. Список спеціальностей порожній або недоступний:
  - 3а.1. Система виводить повідомлення "Наразі немає доступних спеціальностей".
  - 3а.2. Закінчення варіанту використання.
- 8а. Абітурієнт вже має активну заявку на цю спеціальність:
  - 8а.1. Система виводить повідомлення "У вас вже є заявка на цю спеціальність".
  - 8а.2. Система пропонує перейти до редагування існуючої заявки.
  - 8а.3. Закінчення варіанту використання.

### 4. Завантаження документів (A4)

**Контекст використання:** Дає змогу абітурієнту завантажити необхідні скан-копії документів до своєї заявки.

**Зона дії:** Модуль управління документами.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Абітурієнт (авторизований).

**Учасники і інтереси:**
- Абітурієнт - хоче легко завантажити всі необхідні документи
- Система - повинна перевірити формат та розмір файлів
- Відбіркова комісія - хоче отримувати документи у коректному форматі

**Передумова:** Абітурієнт створив заявку і перебуває на сторінці редагування заявки.

**Мінімальні гарантії:** У разі помилки завантаження файл не зберігається, абітурієнт може повторити спробу.

**Гарантії успіху:** Документ успішно завантажено, система відображає його в списку документів заявки.

**Тригер:** Натискання кнопки "Додати документ" або "Завантажити файл".

**Основний сценарій:**
1. Абітурієнт натискає кнопку "Додати документ".
2. Система відображає діалогове вікно вибору типу документа.
3. Абітурієнт обирає тип документа (паспорт, атестат, сертифікат ЗНО/НМТ тощо).
4. Система відкриває діалогове вікно вибору файлу.
5. Абітурієнт обирає файл на своєму пристрої.
6. Система перевіряє формат файлу (PDF, JPG, PNG).
7. Система перевіряє розмір файлу (не більше 5 МБ).
8. Система відображає прев'ю завантаженого документа.
9. Абітурієнт натискає кнопку "Підтвердити завантаження".
10. Система зберігає файл на сервері.
11. Система додає запис про документ до заявки.
12. Система відображає документ у списку завантажених файлів.

**Розширення:**
- 6а. Формат файлу не підтримується:
  - 6а.1. Система виводить повідомлення "Підтримуються лише формати PDF, JPG, PNG".
  - 6а.2. Повернення до кроку 4.
- 7а. Розмір файлу перевищує 5 МБ:
  - 7а.1. Система виводить повідомлення "Розмір файлу не повинен перевищувати 5 МБ".
  - 7а.2. Система пропонує стиснути файл або обрати інший.
  - 7а.3. Повернення до кроку 4.
- 8а. Абітурієнт хоче змінити обраний файл:
  - 8а.1. Абітурієнт натискає кнопку "Обрати інший файл".
  - 8а.2. Повернення до кроку 4.
- 10а. Помилка збереження файлу на сервері:
  - 10а.1. Система виводить повідомлення про технічну помилку.
  - 10а.2. Система пропонує повторити спробу пізніше.
  - 10а.3. Закінчення варіанту використання.

**Основна дійова особа:** Абітурієнт (авторизований).

**Учасники і інтереси:**
- Абітурієнт - хоче знати актуальний статус своєї заявки
- Відбіркова комісія - оновлює статуси після перевірки

**Передумова:** Абітурієнт авторизований і має хоча б одну створену заявку.

**Мінімальні гарантії:** Система відображає останній збережений статус заявки.

**Гарантії успіху:** Абітурієнт бачить актуальний статус заявки та історію змін статусів.

**Тригер:** Перехід до розділу "Мої заявки" в особистому кабінеті.

4. Абітурієнт натискає на конкретну заявку для деталей.
5. Система відображає детальну інформацію про заявку.
6. Система відображає історію зміни статусів із датами.

**Розширення:**

  - 2а.1. Система виводить повідомлення "У вас поки немає заявок".
  - 2а.2. Система пропонує створити нову заявку.
  - 2а.3. Закінчення варіанту використання.

### 6. Перевірка документів (K4)

**Контекст використання:** Дає змогу працівнику відбіркової комісії перевірити коректність, повноту та відповідність документів абітурієнта встановленим вимогам.

**Зона дії:** Модуль роботи з заявками.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Працівник відбіркової комісії.

**Учасники і інтереси:**
- Працівник комісії - хоче ефективно перевірити документи
- Абітурієнт - хоче отримати результат перевірки
- Університет - хоче забезпечити якісний відбір абітурієнтів

**Передумова:** Працівник комісії авторизований і відкрив заявку абітурієнта зі статусом "Подано".

**Мінімальні гарантії:** Статус заявки залишається незмінним до завершення перевірки.

**Гарантії успіху:** Документи перевірені, статус заявки оновлено відповідно до результату перевірки.

**Тригер:** Натискання кнопки "Перевірити документи" на сторінці заявки.

**Основний сценарій:**
1. Працівник комісії відкриває заявку абітурієнта.
2. Система відображає список усіх завантажених документів.
3. Працівник комісії переглядає кожен документ.
4. Працівник комісії перевіряє відповідність документа вимогам.
5. Працівник комісії позначає документ як "Перевірено" або "Потребує виправлення".
6. Після перевірки всіх документів працівник натискає кнопку "Завершити перевірку".
7. Система перевіряє, чи всі обов'язкові документи надані.
8. Система змінює статус заявки на "Перевіряється" або "Прийнято".
9. Система надсилає сповіщення абітурієнту про результат перевірки.

**Розширення:**
- 5а. Документ не відповідає вимогам:
  - 5а.1. Працівник комісії позначає документ як "Потребує виправлення".
  - 5а.2. Працівник комісії додає коментар із поясненням причини.
  - 5а.3. Повернення до кроку 3.
- 7а. Не всі обов'язкові документи надані:
  - 7а.1. Система виводить список відсутніх документів.
  - 7а.2. Система пропонує надіслати запит абітурієнту.
  - 7а.3. Працівник комісії натискає "Надіслати запит".
  - 7а.4. Система змінює статус заявки на "Потребує доповнення".
  - 7а.5. Система надсилає абітурієнту повідомлення зі списком відсутніх документів.
  - 7а.6. Закінчення варіанту використання.
- 7б. Документи містять помилки або невідповідності:
  - 7б.1. Працівник комісії натискає кнопку "Відхилити заявку".
  - 7б.2. Система змінює статус на "Відхилено".
  - 7б.3. Система надсилає абітурієнту детальне пояснення причин відхилення.
  - 7б.4. Закінчення варіанту використання.

### 7. Зміна статусу заявки (K5)

**Контекст використання:** Дає змогу працівнику відбіркової комісії змінити статус заявки абітурієнта відповідно до етапу її розгляду.

**Зона дії:** Модуль роботи з заявками.

**Рівень:** Підфункція.

**Основна дійова особа:** Працівник відбіркової комісії.

**Учасники і інтереси:**
- Працівник комісії - хоче відобразити актуальний стан заявки
- Абітурієнт - хоче бути поінформованим про зміни

**Передумова:** Працівник комісії завершив перевірку документів або прийняв рішення щодо заявки.

**Мінімальні гарантії:** Попередній статус зберігається в історії змін.

**Гарантії успіху:** Статус заявки оновлено, абітурієнт отримав сповіщення.

**Тригер:** Включається автоматично після завершення перевірки документів або натискання кнопки "Змінити статус".

**Основний сценарій:**
1. Система відображає поточний статус заявки.
2. Працівник комісії обирає новий статус зі списку.
3. Система запитує підтвердження зміни статусу.
4. Працівник комісії підтверджує зміну.
5. Система зберігає попередній статус в історії.
6. Система оновлює статус заявки.
7. Система фіксує дату та час зміни статусу.
8. Система надсилає email-сповіщення абітурієнту.

**Розширення:**
- 2а. Вибраний статус недоступний для поточного стану заявки:
  - 2а.1. Система виводить повідомлення про неможливість зміни.
  - 2а.2. Система пояснює дозволені переходи статусів.
  - 2а.3. Закінчення варіанту використання.

### 8. Формування звітів (K7)

**Контекст використання:** Дає змогу працівнику відбіркової комісії генерувати різноманітні звіти для аналізу результатів вступної кампанії.

**Зона дії:** Модуль звітності.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Працівник відбіркової комісії.

**Учасники і інтереси:**
- Працівник комісії - хоче швидко отримати статистичні дані
- Університет - хоче аналізувати ефективність вступної кампанії

**Передумова:** Працівник комісії авторизований у системі.

**Мінімальні гарантії:** У разі помилки генерації звіту виводиться повідомлення про помилку.

**Гарантії успіху:** Звіт згенеровано у вибраному форматі та доступний для завантаження або перегляду.

**Тригер:** Натискання кнопки "Сформувати звіт" у розділі звітності.

**Основний сценарій:**
1. Працівник комісії переходить до розділу "Звіти".
2. Система відображає форму параметрів звіту.
3. Працівник комісії обирає тип звіту (за спеціальностями, за датами, за статусами).
4. Працівник комісії встановлює часовий діапазон.
5. Працівник комісії обирає формат експорту (PDF, Excel, CSV).
6. Працівник комісії натискає кнопку "Сформувати звіт".
7. Система збирає дані згідно з обраними параметрами.
8. Система генерує звіт у вибраному форматі.
9. Система пропонує завантажити або переглянути звіт.

**Розширення:**
- 7а. Немає даних за обраними параметрами:
  - 7а.1. Система виводить повідомлення "За обраними параметрами дані відсутні".
  - 7а.2. Система пропонує змінити параметри.
  - 7а.3. Повернення до кроку 3.
- 8а. Помилка при генерації звіту:
  - 8а.1. Система виводить повідомлення про технічну помилку.
  - 8а.2. Система пропонує повторити спробу або змінити формат.
  - 8а.3. Повернення до кроку 5.

### 9. Управління обліковими записами (AD1)

**Контекст використання:** Дає змогу адміністратору створювати, редагувати та видаляти облікові записи користувачів системи.

**Зона дії:** Адміністративний модуль.

**Рівень:** Цілі користувача.

**Основна дійова особа:** Адміністратор системи.

**Учасники і інтереси:**
- Адміністратор - хоче ефективно управляти користувачами
- Користувачі - хочуть мати коректні облікові записи
- Університет - хоче контролювати доступ до системи

**Передумова:** Адміністратор авторизований з правами адміністратора.

**Мінімальні гарантії:** У разі помилки операції дані користувачів залишаються незмінними.

**Гарантії успіху:** Обліковий запис створено/змінено/видалено відповідно до запиту адміністратора.

**Тригер:** Натискання відповідної кнопки в адміністративній панелі.

**Основний сценарій (створення користувача):**
1. Адміністратор переходить до розділу "Користувачі".
2. Адміністратор натискає кнопку "Створити користувача".
3. Система відображає форму створення облікового запису.
4. Адміністратор заповнює дані: ПІБ, email, роль (абітурієнт, працівник комісії).
5. Адміністратор натискає кнопку "Створити".
6. Система перевіряє унікальність email.
7. Система генерує тимчасовий пароль.
8. Система створює обліковий запис.
9. Система надсилає email з даними для входу на вказану адресу.

**Розширення:**
- 6а. Email вже існує в системі:
  - 6а.1. Система виводить повідомлення "Користувач з таким email вже існує".
  - 6а.2. Повернення до кроку 4.
- 9а. Помилка відправки email:
  - 9а.1. Система виводить попередження про помилку відправки.
  - 9а.2. Система зберігає дані для входу в інтерфейсі адміністратора.
  - 9а.3. Адміністратор може вручну передати дані користувачу.

**Сценарій редагування:**
1. Адміністратор знаходить користувача в списку.
2. Адміністратор натискає кнопку "Редагувати".
3. Система відображає форму з поточними даними.
4. Адміністратор змінює необхідні поля.
5. Адміністратор натискає кнопку "Зберегти зміни".
6. Система зберігає оновлені дані.

**Сценарій видалення:**
1. Адміністратор знаходить користувача в списку.
2. Адміністратор натискає кнопку "Видалити".
3. Система запитує підтвердження видалення.
4. Адміністратор підтверджує видалення.
5. Система перевіряє наявність активних заявок користувача.
6. Система деактивує обліковий запис (не видаляє, щоб зберегти історію).

**Учасники і інтереси:**
- Адміністратор - хоче забезпечити збереження даних

**Передумова:** Адміністратор авторизований з правами адміністратора.

**Основний сценарій:**

### 11. Експорт даних до ЄДЕБО (S1)

**Контекст використання:** Дає змогу системі автоматично передавати дані про абітурієнтів та їх заявки до державної системи ЄДЕБО відповідно до законодавчих вимог.

**Зона дії:** Модуль інтеграції з зовнішніми системами.

**Рівень:** Системна функція.

**Основна дійова особа:** Система (автоматична операція).

**Учасники і інтереси:**
- Система ЄДЕБО - хоче отримувати актуальні дані у встановленому форматі
- Університет - повинен виконувати законодавчі вимоги
- Відбіркова комісія - хоче автоматизувати процес передачі даних

**Передумова:** Заявка абітурієнта має статус "Прийнято" та містить усі необхідні дані.

**Мінімальні гарантії:** У разі помилки передачі дані зберігаються у черзі для повторної спроби.

**Гарантії успіху:** Дані успішно передано до ЄДЕБО, отримано підтвердження.

**Тригер:** Зміна статусу заявки на "Прийнято" або за розкладом (щоденно о 23:00).

**Основний сценарій:**
1. Система виявляє заявки зі статусом "Прийнято", які не передані до ЄДЕБО.
2. Система формує пакет даних у форматі XML згідно зі специфікацією ЄДЕБО.
3. Система перевіряє валідність сформованого XML.
4. Система встановлює захищене з'єднання з ЄДЕБО.
5. Система передає дані до ЄДЕБО.
6. Система очікує на підтвердження отримання.
7. ЄДЕБО повертає статус "Прийнято" та унікальний ідентифікатор.
8. Система зберігає ідентифікатор ЄДЕБО у записі заявки.
9. Система встановлює прапорець "Передано до ЄДЕБО".
10. Система записує інформацію про передачу до логу.

**Розширення:**
- 3а. XML не пройшов валідацію:
  - 3а.1. Система записує помилку до логу.
  - 3а.2. Система надсилає сповіщення адміністратору.
  - 3а.3. Система залишає заявку у черзі для ручної перевірки.
  - 3а.4. Закінчення варіанту використання.
- 4а. Неможливо встановити з'єднання з ЄДЕБО:
  - 4а.1. Система записує помилку до логу.
  - 4а.2. Система робить повторну спробу через 15 хвилин.
  - 4а.3. Після 3 невдалих спроб система надсилає сповіщення адміністратору.
- 7а. ЄДЕБО повертає статус помилки:
  - 7а.1. Система аналізує код помилки.
  - 7а.2. Система записує детальний опис помилки до логу.
  - 7а.3. Система надсилає сповіщення працівнику комісії для виправлення даних.
  - 7а.4. Заявка залишається у черзі для повторної передачі після виправлень.

## Висновки

## Відповіді на контрольні питання

### 1. Дайте визначення вимогам користувача

Вимоги користувача — це опис потреб, очікувань та цілей кінцевих користувачів системи. Вони формулюють, що саме користувач хоче отримати від програмного продукту, без деталізації технічної реалізації.

### 2. Назвіть джерела вимог користувача

- Інтерв’ю з користувачами та замовниками.
- Анкетування та опитування.
- Спостереження за робочими процесами.
- Аналіз існуючих систем та їх недоліків.
- Документи та нормативні акти (наприклад, державні стандарти).
- Прототипи та тестові версії продукту.

### 3. Назвіть методи виявлення вимог

Серед методів виявлення вимог: інтерв'ю, анкетування, спостереження, brainstorm, прототипування, аналіз документів і бізнес-процесів та JAD-наради.

### 4. Дайте визначення акторові

Актор - це зовнішній учасник системи (користувач або інша система), який взаємодіє з програмним продуктом і виконує певні дії.

### 5. Що таке варіант використання?

Варіант використання - це опис конкретної взаємодії актора із системою задля досягнення конкретно визначеної мети.

### 6. Що таке UML?

Це уніфікована мова моделювання, яка використовується для опису, візуалізації та документування програмних систем. UML включає діаграми класів, варіантів використання, послідовностей, діяльності тощо.

### 7. Які відношення застосовують в цих діаграмах? Поясніть їх на прикладах

- Include (включення): коли один варіант використання обов'язково включаї інший.
- Extend (розширення): додатковий варіант використання, що виконується за певних умов.
- Generalization (узагальнення): коли актор або юзкейс є спеціалізацією іншого. Наприклад, Абітурієнт та Працівник комісії є спеціалізацією актора Користувач.

### 8. Що таке розширений опис вимог?

Це деталізований опис функціональних та нефункціональних вимог, який включає контекст використання, передумови, основний сценарій, альтернативні сценарії та післяумови.

### 9. Які типи такого опису Ви знаєте? Опишіть їх

- Функціональні вимоги: описують, що повинна робити система, який функціонал мати.
- Нефункціональні вимоги: описують якість системи (швидкодія, безпека, надійність, зручність тощо).
- Бізнес-вимоги: описують цілі організації, яку повинна підтримувати система.
- Системні вимоги: це технічні обмеження та умови (сумісність з браузерами, інтеграція з іншими системами тощо).
